# 블루스캔 타깃 공략 에이전트 (실습 버전)

에스원 영업 컨설턴트가 블루스캔(원격 건물 관리 솔루션) 방문 타깃을 정하는 것을 돕는 도구.
공공·관제 예시 데이터를 규칙 기반으로 점수화해 우선순위 리스트로 제시한다.

Claude Design 목업(`../에스원 디자인 (4)`)을 **React + Vite** 앱으로 포팅한 것이다.

## 실행

```bash
cd app
npm install
npm run dev      # http://localhost:5173
npm run build    # dist/ 정적 빌드
npm run preview  # 빌드 결과 미리보기
```

## 주요 화면

- **대시보드** — 역할(영업사원/관리자) 전환. 발굴 후보·방문 접촉·파이프라인·지역 밀집도 히트맵.
- **신규 고객 후보 (리스트 A)** — 건축물대장·노후건축물 기반 도입 가능성 점수 + 화재 가산점.
  지역/용도 필터, 점수·노후화·규모순 정렬, Leaflet 지도(후보 마커 + 화재 오버레이), 점수 근거 펼치기.
- **업셀링 후보 (리스트 B)** — 사내 관제(index) 키워드 매칭 기반 업셀링 적합도 점수와 매칭 근거.
- **유지고객 대시보드** — 이미 블루스캔을 쓰고 있는 계약 고객(테마 B, 리스트 B와 무관)의 관제 신호·계약
  현황. 이탈 스코어링 없이 신호를 있는 그대로 보여준다(1차 버전).
- **방문 결과 기록** — 후보별 방문 결과 입력(음성 받아쓰기 + AI/휴리스틱 분석), 상태·메모, CSV 다운로드.

두 리스트는 척도가 달라 **합산하지 않고 별도 화면**으로 유지한다. 데이터가 없는 항목은 0점이 아니라
**NO_DATA**로 구분하며, 멸실·중복 건은 후보에서 제외 표시한다. 점수는 우선순위 참고 신호이며 최종 판단은 담당자가 한다.

## 기술 노트

- **디자인 시스템**: 목업의 S-1 디자인 시스템 번들(`_ds_bundle.js`)을 그대로 사용한다. 이 번들은
  클래식 `React.createElement`로 컴파일되어 전역 `React`를 참조하므로, `main.jsx`에서 렌더 전에
  `window.React`를 노출하고 번들은 `index.html`에서 모듈 로드 전에 classic script로 불러온다.
- **데이터**: `window.APPDATA`(예시 시드). 백엔드·DB 없음(local-first). 방문 결과는 세션 메모리 +
  CSV 내보내기. 민감정보·API Key는 포함하지 않는다.
- **AI 분석**: 방문 대화 분석은 호스트의 `window.claude`가 있으면 사용하고, 없으면 로컬 휴리스틱으로
  폴백한다(외부 호출·키 불필요).
- **화재 출동 현황(목업)**: 컨설턴트 대시보드의 "블루스캔 관련 뉴스" 카드에 소방청 화재정보 공개API
  (국민안전24) 기반 전국 요약 배지(접수/진행/인명/재산피해)와, safekorea 실시간 화재정보 형태의
  **개별 출동 건 스크롤 리스트**(`fireDispatch.demo.js`의 `liveItems`)를 표시한다. 시도별 집계 막대는
  영업 관점에서 의미가 약해 제외했다. 같은 `liveItems`를 리스트 A·B 지도에도 마커(🚒)로 오버레이하며,
  기존 "화재 오버레이" 토글 한 번으로 화재 발생 구역(점선)과 실시간 출동 지점이 함께 켜지고 꺼진다.
  실연동 시 `fireDispatch.demo.js`의 `buildFireDispatchDemo()` 반환 구조(national/bySido/liveItems)
  그대로 API 응답을 채우면 화면 변경 없이 교체된다.
- **시즌별 특화 타깃(1차: 침수 위험 신호)**: 건축물대장 후보(`listA`)는 빌드 시점에 이미 도시침수
  예상구역(기왕최대) 폴리곤과의 point-in-polygon 판정 결과(`c.flood = { level, area }`)를 갖고 있다.
  이 신호를 혹서기·풍수해기(6~9월)에만 노출하도록 `season.js`의 `currentSeasonKey()`로 게이팅했다
  (그 외 기간엔 배지·알림·대시보드 카드가 숨겨짐). 컨설턴트 대시보드에 "시즌 타깃 신호" 카드를 추가해
  시즌별로 위험 후보 목록을 보여주며, 카드 우측 선택창으로 실제 날짜와 무관하게 시즌을 미리보기할 수 있다
  (데모/QA 용도). 점수(`score`)에는 반영하지 않고 영업 타이밍 신호로만 사용한다(PRD 원칙 유지).
- **유지고객(리스트 B) 침수 위험 + 지하 중요시설 세분화**: 리스트 B는 원본 데이터에 침수 판정이
  없었으므로, 이미 앱에 실려있는 실제 침수 폴리곤(`window.APPDATA.floodLayers[].geo`)을 그대로
  재사용해 `floodRisk.js`가 클라이언트에서 동일한 point-in-polygon 방식으로 판정한다(실제 지오메트리
  기반 — 목업 아님). 다만 침수 위험 + B-2(중요실 보유) 고객처에 붙는 "지하 전기실(수배전반)/지하
  펌프실/지하 발전기실" 등 세부 태그는 원본 데이터에 없는 항목이라 id 기반으로 결정론적 예시 태그를
  부여한 것이다(`demoUndergroundRoomFor`) — 실연동 시 이 태그만 실제 현장조사·관제 데이터로
  교체하면 된다. 대시보드 "시즌 타깃 신호" 카드와 리스트 B 화면 배지·상세패널에 함께 표시된다.
- **유지고객(블루스캔) 대시보드 — 테마 B 1차 버전**: 리스트 B(업셀링 대상 — 블루스캔 미도입 고객)와는
  완전히 별개로, 이미 블루스캔을 쓰고 있는 계약 고객을 다루는 신규 탭이다. 블루스캔 서비스 구조
  (https://www.s1.co.kr/estate/manage/smart-building — 오너/듀얼 티어, 이상발생 5종 신호: 화재·정전·
  가스누출·누수·고저수위)를 반영해 `retentionSchema.js`에 필드를 정의하고, `retention.demo.js`가
  40건을 생성한다. 시연 계정 범위에 맞춰 **전건을 서강지사(마포구) 관할로 한정**했다(마포구 내 12개
  동을 순환시켜 지역 다양성만 확보 — 다른 지사 데이터는 없음). 듀얼 80%/오너 20% 비율.
  화면 맨 위에 다른 대시보드와 같은 형태의 "요약 알림" 카드를 두어, 전 계약처 신호 발생 내역(일시·
  계약처명·신호종류, 최신순)과 계약 만료 임박(60일 이내) 목록을 2열로 나눠 각각 최대 180px
  스크롤 영역에 담았다(카드가 무한정 길어지지 않도록). 신호 이력에는 "미해결" 여부를 두지 않는다
  (요청에 따라 제외) — 심각도(경미/주의/심각)만 표시한다. 사이드바 메뉴에도 다른 리스트처럼
  전체 유지고객 수 배지가 붙는다. 대시보드 항목(요약 알림/핵심 지표/목록·지도)은 `Dash.jsx`에서
  export한 `useSectionOrder`/`SectionList`를 그대로 재사용해 다른 대시보드와 동일한 방식(좌상단
  드래그 핸들, "구성 초기화" 버튼, localStorage 순서 저장)으로 순서를 바꿀 수 있다.
  **월간 리포트 자동생성(R23) — PDF**: 고객 상세를 펼치면 "초안 생성" 버튼으로 `monthlyReport.js`가
  신호 내역·원격제어 이력(둘 다 실데이터, `remoteControlLog` 신규)·최근 인근 사건사고를 조합해
  리포트를 만든다. 인근 사건사고는 (1) 화재 출동 데이터(`fireDispatch.demo.js`)와 (2) **인근 다른
  계약처의 최근 신호를 계약처명·정확한 주소 없이 동(洞) 단위로만 익명화해 공유**하는 두 가지를
  함께 보여준다("다른 건물은 이런 일도 있었다"). 화면 미리보기(`ReportPreview`)를 `html2canvas`로
  캡처해 `jsPDF`로 실제 PDF 파일을 만든다(`pdfExport.js`) — jsPDF 기본 폰트는 한글이 깨지므로
  브라우저가 렌더링한 화면을 이미지로 캡처하는 방식으로 우회했다. "승인 후 발송 처리" 버튼은
  실제 발송 없이 발송일자만 화면에 반영하는 모의 동작이다(PRD 원칙: 실제 발송은 사람 승인 필요).
  리포트는 통신사·카드사 청구서 느낌으로 팝업(`RDialog`)에 띄운다 — 그라디언트 요약 배너(이번달
  신호·원격제어 사용·계약 잔여·생애가치 상태 4개 히어로 카드), 신호 유형 도넛 차트, 색상 액센트가
  있는 섹션 카드(청구서의 "이용내역"과 유사), 대시된 구분선의 항목 리스트로 구성했다(`app.css`의
  `.mreport*` 클래스). 팝업은 `createPortal`로 `document.body`에 직접 띄운다 — 목록 영역 안에서
  렌더링하면 부모 스크롤 컨테이너에 갇혀 리스트와 같이 스크롤되는 문제가 있었다. 빌드는 통과했지만
  PDF 캡처 자체는 실제 브라우저 클릭으로 최종 확인이 필요하다(이 환경엔 headless 브라우저가 없다).
  **고객사 제출용 리포트에는 생애가치(BEP/ROI·할인율 등 내부 수익성 정보)를 포함하지 않는다** —
  그 정보는 컨설턴트 전용 화면(고객 상세의 `LifetimeValueBox`)에만 남겨뒀다.
- **감성터칭 메시지(R25)**: 고객 상세의 "관제 신호 이력" 각 행에 하트 아이콘 버튼을 추가했다.
  실제 서비스 흐름상 이상 발생 시 관제사가 즉시 1차 연락을 하므로, 이 메시지는 **영업담당자가
  사후(1~3일 뒤) 보내는 안부 팔로업**이다("화재가 감지됐습니다" 같은 통보문이 아니다). 누르면
  `empathyMessage.js`가 (1)놀라진 않으셨는지 (2)잘 해결됐는지 (3)최근에도 있었는지·처음인지
  (4)다른 이슈 있으면 연락 달라는 흐름의 문안을 자동 생성해 팝업으로 보여주고, 담당자가
  textarea에서 직접 수정한 뒤 "승인 발송(모의)"을 누르면 고객 상세의 "감성터칭 최근 발송"
  날짜가 갱신된다(실제 발송 없음, PRD 원칙: 발송은 사람 승인).
- **생애가치(BEP/ROI) — 유지고객 데이터 고도화**: 공사비 할인율(`installCostDiscountRate`)·정가
  (`standardInstallCost`)를 추가하고, 손익분기(BEP, `bepMonths`/`bepReached`)·3년 계약기준 ROI
  (`roi3yr`)·누적 순가치(`netValueToDate` = 월서비스료×유지개월 − 공사비)를 더미로 계산해 넣었다.
  `netValueToDate`가 음수면 "지금 해약되면 회사가 손해 보는 상태 — 할인해서라도 유지 필요", 양수면
  "이미 손익분기를 넘겨 해약 시즌 할인 여력이 있는 고객"이라는 식으로 화면에 바로 해석까지 붙여
  보여준다(가중치 스코어링이 아니라 그대로 계산된 재무 수치).
  **이탈 스코어·등급은 이번 버전에 없다** —
  PRD상 예측 모델 기준은 사람이 설계해야 하는 영역이라, 신호(미해결 여부·계약 만료 임박·VOC)를
  그대로 보여주고 "주의 필요" 여부만 규칙 기반(스코어 아님)으로 표시한다. 지도는 `map.jsx`에 신규
  variant `'C'`를 추가해 재사용했다(주의필요=빨강/안정=파랑, 점수 티어 없음).
- **게이미피케이션(엑셀 "B.T.S Quest" 아이디어 반영)**: 컨설턴트 대시보드에 "이번 주 퀘스트·랭킹"
  카드를 추가했다. `gamification.js`가 앱에 이미 있는 상태(방문 결과, 유지고객 리포트·감성터칭
  발송 여부)를 그대로 읽어 포인트·퀘스트 진행률·배지·랭킹을 계산하는 파생 뷰다 — 별도 DB나
  이벤트 로그를 새로 만들지 않았다. 리더보드는 로그인한 본인은 실제 세션 포인트, 같은 지사
  동료는 기존 `consultantPerf`와 같은 사번 시드 방식의 예시 점수를 합쳐서 보여준다(동료 점수는
  실데이터 아님). 포인트 배점(`POINTS`)과 퀘스트 난이도(`QUEST_DEFS`)는 예시 숫자이며 실제
  운영 시 영업 관리자가 검토·확정해야 한다. 처음엔 이 카드가 컨설턴트 대시보드 안에만 있어서
  "행동해도 변화가 안 보인다"는 문제가 있었다 — 지금은 상단바(`topbar__points`)에 포인트 배지를
  항상 띄우고, 포인트가 오르는 순간 "+N" 애니메이션(`points-pop`)이 뜨도록 고쳐서 어느 화면에
  있든(방문 결과 입력 팝업이 열려 있어도) 바로 체감되게 했다. 결과를 저장하는 그 순간에는
  `toastMessages.js`가 상황별(방문·수주·타이밍대응·리포트·감성터칭·배지획득·퀘스트완료)로 다른
  다정한 응원 문구를 골라 화면 우측 하단에 팝업(`gm-toast`)으로 띄운다 — 아이콘이 살짝 통통
  튀는 애니메이션(`toast-icon-bounce`)이 있어 딱딱한 시스템 알림투가 아니게 했다. 앱 실행 시
  이미 깔려있는 데모 시드 데이터(방문 3건 등)에 대해서는 팝업이 뜨지 않도록, 각 항목의 "이전
  상태"를 마운트 시점에 한 번 기록해두고 그 이후의 변화만 팝업 대상으로 잡는다.
