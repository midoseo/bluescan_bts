# 건축물대장 실연동 개발 가이드

## 0. 목적

신규진행현황의 **기존 고객(업셀) 물건**(`listB` = 서강지사 계약 물건)은 지금 주소·경비원(b1)·전산실(b2, 더미)만 있고, "건축물대장 우량물건 겹침(=최고 타깃)"은 `btype='both'` **모델 플래그**로만 표시돼 있다. 이 가이드는 그 모델 플래그를 **실제 건축물대장 데이터**로 대체하는 연동 설계다.

물건 주소 → 건축물대장 조회로 **연면적·층수·사용승인일(노후)·구조·주용도**를 실제로 붙이면, 신규 후보(`listA`)와 **동일한 규칙 기반 스코어링**을 기존 물건에도 적용할 수 있고, "우량 물건" 판정을 근거 기반으로 할 수 있다.

---

## 1. 전체 데이터 흐름

```
서강 물건 주소(계약원장)
   │  ① 주소 정제 (동/호 제거, 지번↔도로명 판별)
   ▼
도로명주소 API(주소검색)            ← 키 A (juso.go.kr)
   │  ② admCd(법정동코드10) · 지번본번/부번 획득
   ▼
코드 분해 (sigunguCd·bjdongCd·bun·ji)
   │  ③
   ▼
건축물대장 표제부 조회 API          ← 키 B (data.go.kr 건축HUB)
   │  ④ 연면적·층수·사용승인일·구조·주용도
   ▼
앱 데이터 매핑 (listB 물건에 buildingLedger 필드 추가)
   │  ⑤ 스코어 가산 + "건축물대장 매칭" 배지
   ▼
신규진행현황(Pipeline) 렌더
```

핵심: **주소만으로는 대장 조회가 안 된다.** 건축물대장 API는 `시군구코드+법정동코드+지번(본번·부번)`을 키로 받으므로, 먼저 주소를 이 코드로 변환하는 단계(②③)가 반드시 필요하다.

---

## 2. 발급받아야 할 키 (서지나님 담당)

| 구분 | 서비스 | 발급처 | 용도 |
| --- | --- | --- | --- |
| **키 A** | 도로명주소 개발자센터 — 주소검색 API | business.juso.go.kr | 주소 → 법정동코드·지번 |
| **키 B** | 국토교통부 건축HUB_건축물대장정보 서비스 | data.go.kr/data/15134735 | 법정동코드·지번 → 대장정보 |
| (선택) | 도로명주소 좌표제공 API 또는 브이월드 | juso.go.kr / vworld.kr | 지도 핀 좌표 정밀화(현재는 동 단위 근사) |

- 두 API 모두 **무료·집계정보**라 개인정보 이슈 없음(단, 건축물 *소유자* 조회 서비스는 별도이며 이번 범위에서 제외 — 개인정보 포함).
- 공공데이터포털 API는 신청 후 보통 1~2시간 내 활성화(일부 24시간).
- 트래픽 한도(일일 호출수) 확인 필요 — 서강 물건은 약 100건(경비원·전산실 보유분)이라 **1회 배치 처리로 충분**하다.

## 3. 단계별 상세

### ① 주소 정제
서강 물건 주소 예시:
- `서울 서대문구 경기대로58, 3층 (충정로2가,경기빌딩)` → 도로명 + 층 + (법정동,건물명)
- `서울 마포구 공덕2동457번지 래미안3차 305동 1904호` → 지번 + 동/호

처리:
- **동/호·층 제거**: `\d+동`, `\d+호`, `\d+층`, `B\d+` 등 상세 단위 제거(건물 단위로 정규화).
- **도로명/지번 판별**: "로/길 + 숫자"면 도로명, "동/리 + 숫자번지"면 지번.
- 괄호 안 `(법정동,건물명)`에서 **법정동·건물명**을 보조 키로 확보(매칭 검증용).

### ② 도로명주소 API — 주소검색

```
GET https://business.juso.go.kr/addrlink/addrLinkApi.do
  ?confmKey={키A}
  &currentPage=1&countPerPage=5
  &keyword={정제한 주소}
  &resultType=json
```

응답에서 확보할 필드:

| 필드 | 의미 | 예 |
| --- | --- | --- |
| `admCd` | 법정동코드(10자리) | `1141010100` |
| `lnbrMnnm` | 지번 본번 | `457` |
| `lnbrSlno` | 지번 부번 | `0` |
| `bdMgtSn` | 건물관리번호(24자리, 매칭 검증용) | … |
| `roadAddr`/`jibunAddr` | 정규화된 도로명/지번 주소 | … |

> 결과가 여러 건이면 ①에서 확보한 **건물명·법정동**으로 최적 후보를 고른다. 0건이면 `NO_DATA`로 표시(뒤 5장 예외처리).

### ③ 코드 분해

```
sigunguCd = admCd.slice(0, 5)     // 예: 11410 (서대문구)
bjdongCd  = admCd.slice(5, 10)    // 예: 10100 (충정로2가)
platGbCd  = "0"                    // 0:대지 1:산 2:블록
bun       = String(lnbrMnnm).padStart(4, "0")  // 예: "0457"
ji        = String(lnbrSlno || 0).padStart(4, "0")
```

### ④ 건축물대장 표제부 조회 (건축HUB)

일반 건물은 **표제부(getBrTitleInfo)**, 아파트 등 집합건물 단지는 **총괄표제부(getBrRecapTitleInfo)**를 쓴다. 우량물건 판정에 필요한 핵심값은 표제부에 다 있다.

```
GET https://apis.data.go.kr/1613000/BldRgstHubService/getBrTitleInfo
  ?serviceKey={키B}
  &sigunguCd={sigunguCd}&bjdongCd={bjdongCd}
  &platGbCd=0&bun={bun}&ji={ji}
  &_type=json&numOfRows=100&pageNo=1
```

응답 `response.body.items.item[]`에서 쓸 필드:

| 앱 필드 | 대장 필드(영문) | 의미 |
| --- | --- | --- |
| `gfa` | `totArea` | 연면적(㎡) |
| `grndFlr` | `grndFlrCnt` | 지상 층수 |
| `ugrndFlr` | `ugrndFlrCnt` | 지하 층수 |
| `approvalDate` | `useAprDay` | 사용승인일(YYYYMMDD) |
| `struct` | `strctCdNm` | 구조(철근콘크리트 등) |
| `mainUse` | `mainPurpsCdNm` | 주용도(업무시설·교육연구시설 등) |
| `bldNm` | `bldNm` | 건물명(검증용) |

- **노후연수** = 올해 − `useAprDay` 연도 → `approvalYrAgo`.
- 한 지번에 동이 여럿이면 item이 여러 개 → 연면적 최대 동 또는 주건축물(`mainAtchGbCdNm='주건축물'`) 기준으로 대표값 선택.

### ⑤ 앱 데이터 매핑 + 스코어 반영

각 서강 물건에 `buildingLedger`를 붙인다:

```jsonc
{
  "id": "SGB3000",
  "name": "…",
  "address": "…",
  "b1": true, "b2": false,
  "buildingLedger": {
    "matched": true,
    "gfa": 5231.4, "grndFlr": 12, "ugrndFlr": 2,
    "approvalDate": "1998-05-20", "approvalYrAgo": 28,
    "struct": "철근콘크리트구조", "mainUse": "업무시설",
    "bdMgtSn": "…"        // 매칭 근거
  }
}
```

`upsellScore.js`에 **건축물대장 가산 항목**을 추가(예: max 15점) — 규모(연면적)·노후·지하층 보유 등 `listA`의 `comps`와 같은 기준을 재사용:

```js
// upsellScore.js — 5) 건축물대장 우량도 (max 15)
const bl = c.buildingLedger;
if (bl?.matched) {
  let v5 = 0; const n5 = [];
  if (bl.gfa >= 10000) { v5 += 6; n5.push('연면적 1만㎡↑'); }
  else if (bl.gfa >= 3000) { v5 += 4; n5.push('연면적 3천㎡↑'); }
  if (bl.approvalYrAgo >= 20) { v5 += 5; n5.push('20년 이상 노후'); }
  if (bl.ugrndFlr >= 1) { v5 += 4; n5.push('지하층 보유(침수·설비 리스크)'); }
  v5 = Math.min(15, v5);
  comps.push({ k: '건축물대장 우량도', v: v5, max: 15, note: n5.join(' · ') || '해당 없음' });
}
```

- 현재 `btype==='both'`로 표시하던 "최고 타깃"은 **`buildingLedger.matched && (b1||b2) && 우량도 높음`** 조건으로 대체.
- Pipeline 행 배지에 **"건축물대장 매칭"**(초록) 추가, 미매칭은 `NO_DATA` 표기(0점 아님).

## 4. 반영 방식 — 배치 전처리 권장

서강 물건은 계약원장 기준으로 **고정**이므로, 앱 런타임에서 매번 외부 API를 부르지 말고 **1회 배치**로 대장정보를 붙여 `pipelineSeogang.generated.js`를 갱신하는 게 안전하다.

```
build-buildingledger.mjs (신규 배치 스크립트)
  1. pipelineSeogang.generated.js의 SEOGANG_LISTB 로드
  2. 각 물건 주소 → ②③④ 수행 (juso → 건축물대장)
  3. buildingLedger 필드 병합
  4. 파일 재작성 (SEOGANG_LISTB 갱신)
  + 호출 결과를 buildingLedger.cache.json에 캐시(재실행 시 재호출 방지)
```

- 갱신 주기: 계약 물건 변동 시(월 1회 등)만 재실행.
- 실시간이 꼭 필요하면(신규 물건 즉시 조회) 서버리스 프록시(Netlify Functions 등)에서 키를 숨기고 호출 — **키를 프론트엔드에 노출 금지**.

## 5. 예외·품질 처리

| 상황 | 처리 |
| --- | --- |
| 주소검색 0건 | `buildingLedger.matched=false` → 화면 `NO_DATA`(0점 아님, 합산 제외) |
| 지번에 동 여러 개(집합/단지) | 주건축물·연면적 최대 동 대표값 사용, 또는 총괄표제부 조회 |
| 도로명↔지번 불일치 | juso가 반환한 `jibunAddr`로 재검증, 건물명 대조 |
| 층/호 단위 중복 물건 | ①에서 건물 단위로 dedup 후 1회만 조회 |
| API 한도 초과 | 캐시 + 배치 분할(일 단위) |
| 응답 지연/오류 | 기존 모델 플래그로 폴백(에러 없이 조용히), 갱신 실패일 로깅 |

## 7. 개발자 전달용 요약

1. `build-buildingledger.mjs` 배치 작성: 주소→juso→건축물대장 표제부→`buildingLedger` 병합→`pipelineSeogang.generated.js` 갱신(+캐시).
2. `upsellScore.js`에 "건축물대장 우량도(max 15)" 항목 추가.
3. `Pipeline.jsx` 행에 "건축물대장 매칭" 배지 + 상세 comps에 우량도 표시, 미매칭 `NO_DATA`.
4. `btype==='both'`(모델) → `buildingLedger.matched` 실판정으로 최고 타깃 기준 교체.
