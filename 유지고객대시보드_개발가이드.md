# 유지고객 대시보드 개발 가이드

리스트 B(업셀링 — 블루스캔 **미도입** 고객)와는 완전히 별개인 신규 탭이다. 이미 블루스캔을 쓰고 있는 계약 고객(테마 B: 리텐션 보강 대상)의 관제 신호·계약 현황·월간 리포트를 다룬다. 이 문서는 이번에 실제로 만든 파일 전체와, 더미 데이터를 실데이터로 바꾸는 방법을 정리한다.

## 1. 만든 것 요약

- 사이드바에 "유지고객 대시보드" 탭 신규 추가(건수 배지 포함)
- 요약 알림(신호 발생 내역 + 계약 만료 임박) · 핵심 지표(KPI) · 목록+지도 3개 섹션, 다른 대시보드와 동일한 방식으로 드래그 재정렬 가능
- 유지현황 지도(주의필요=빨강 / 안정=파랑 핀, 스코어링 아님)
- 고객 상세: 계약 정보, 설비 구성, 관제 신호 이력, 활성도(원격제어·VOC·리포트발송), **생애가치(BEP/ROI, 컨설턴트 전용)**
- **월간 리포트**(테마 B, R23): AI가 초안 생성 → 청구서 스타일 팝업 미리보기 → PDF 다운로드 → "승인 후 발송 처리(모의)". 고객사 제출 자료라 생애가치 정보는 제외했다.
- **감성터칭 메시지**(테마 B, R25): 관제 신호 이력 각 행에 버튼 → AI가 안부 메시지 초안 생성 → 담당자가 텍스트 수정 → "승인 발송(모의)". 아래 4-5 참고 — 실제 서비스 흐름을 반영해 "즉시 통보"가 아니라 "사후 팔로업"으로 설계했다.

## 3. 데이터 스키마 (`retentionSchema.js`)

블루스캔 실제 서비스 구조(https://www.s1.co.kr/estate/manage/smart-building — 오너/듀얼 티어, 이상발생 5종: 화재·정전·가스누출·누수·고저수위)를 반영했다.

| 그룹 | 필드 |
| --- | --- |
| 기본 식별·계약 | `name, address, sido/sigungu/gun/dong, use(업종 5종), branch, lat/lng, contractNo, contractDate, startDate, endDate, contractMonths, monthlyFee, installCost, installCostDiscountRate, standardInstallCost` |
| 서비스 티어·설비 | `productTier(owner/dual), sensorTypes[], remoteControlDevices[]` |
| 신호·이력 | `signalHistory[]{date,type,severity,notifiedAuthority}, signalCount30d/90d, signalTrend, remoteControlLog[]{date,device,action}, remoteControlUsage30d` |
| 활성도·소통 | `lastAppAccessDate, unresolvedVOC, monthlyReportSent, lastTouchDate, assignedConsultant` |
| 생애가치(BEP/ROI) | `bepMonths, bepReached, roi3yr, netValueToDate` — 계산식 그대로 실제 로직(뒤 4-3 참고) |

이탈 스코어·등급(`churnScore`/`churnRisk`)은 **의도적으로 없음** — PRD 원칙상 예측 모델 기준은 사람이 설계해야 하는 영역. "주의 필요" 배지는 계약만료 60일 이내·심각 신호 존재·미해결 VOC 중 하나만 해당하면 뜨는 단순 규칙(`Retention.jsx`의 `needsAttention()`)이지 가중치 스코어가 아니다.

## 4. 더미 데이터를 실데이터로 바꾸는 방법

### 4-1. 계약·설비 정보
`retention.demo.js`의 `buildRetentionDemo()`를 통째로 교체 대상으로 본다. 실제 계약관리시스템/ERP에서 계약번호·계약일·개시일·종료일·월서비스료·공사비·할인율·설치 센서·원격제어 기기 목록을 가져와 `retentionSchema.js`의 필드 구조 그대로 채우면 된다. **좌표(lat/lng)는 정밀 지오코딩이 필요**하다 — 지금은 서강지사(마포구) 12개 동 중심점 기준으로 배치한 것이라, 실제 주소 기반 좌표로 교체하면 지도(`variant="C"`)는 코드 수정 없이 그대로 동작한다.

### 4-2. 관제 신호·원격제어 이력
`signalHistory`는 실제 관제센터(블루에셋) 시스템의 이상감지 로그로, `remoteControlLog`는 앱/웹 원격제어 액션 로그로 교체한다. 필드 구조(`date/type/severity/notifiedAuthority`, `date/device/action`)만 맞추면 `signalCount30d/90d`·`signalTrend`·`remoteControlUsage30d` 계산 로직(`retention.demo.js` 안)도 실데이터에 그대로 적용된다 — 이 계산 자체는 목업이 아니라 실제 로직이다.

### 4-3. 생애가치(BEP/ROI) — 계산식은 이미 실제 로직, 입력값만 실데이터로
```
standardInstallCost = installCost / (1 - installCostDiscountRate/100)
bepMonths           = installCost / monthlyFee
roi3yr(%)           = (monthlyFee*36 - installCost) / installCost * 100   // 36 = CONTRACT_TERM_MONTHS(3년)
netValueToDate       = monthlyFee*contractMonths - installCost
```
실계약 데이터(4-1)만 정확히 들어오면 이 4개 값은 자동으로 정확해진다 — **이 부분은 코드를 고칠 필요가 없다.**

### 4-4. 인근 사건사고(월간 리포트용)
- 화재: 이미 만든 `화재_국민안전24연동_개발가이드.md` 참고(같은 `fireDispatch.demo.js` 재사용).
- 인근 계약처 신호(익명화): `monthlyReport.js`의 `buildAnonymizedNearbySignals()`가 `allCustomers` 배열(=전체 유지고객 목록)에서 같은 시군구를 찾아 계약처명·정확 주소 없이 동(洞) 단위로만 추려낸다. `retention.demo.js`가 실데이터로 교체되면 이 로직도 수정 없이 그대로 작동한다.

### 4-5. 감성터칭 메시지 — 실제 서비스 흐름 주의사항
**이상 발생 시 관제사가 고객에게 즉시 1차 연락을 한다(이미 완료된 절차).** 이 기능은 그 통보가 아니라, **영업담당자(컨설턴트)가 사후 1~3일 뒤 보내는 안부 팔로업**이다. 그래서 문구가 "화재가 감지됐습니다"가 아니라 "관제팀에서 연락드렸던 것으로 아는데, 놀라진 않으셨는지/잘 해결됐는지/최근에도 있었는지·처음인지" 순서로 구성돼 있다. 이 흐름을 다른 방향(예: 실시간 알림)으로 바꾸려 하면 실제 관제 프로세스와 어긋나니, 화면·문구를 고칠 때 이 전제를 유지해야 한다.

현재 `buildEmpathyMessageDraft()`는 **고정 템플릿**(신호 유형에 따라 문구 한 단어만 바뀜)이라 PRD가 말하는 "AI가 문안 생성"의 가장 단순한 형태다. 실연동 시 다음 중 하나로 고도화한다.
1. 템플릿을 유지하되 문구 다양성만 늘림(간단, AI 호출 불필요)
2. `Visit.jsx`의 AI 분석 방식(`window.claude.complete` 또는 백엔드 `/api/*` 프록시)처럼 실제 LLM 호출로 교체 — 신호 맥락(유형·심각도·고객명·발생일)을 프롬프트에 넣어 매번 새로운 문안을 생성하게 할 수 있다.

## 5. 월간 리포트 팝업 관련 메모

- **팝업이 리스트와 같이 스크롤되던 버그**: 목록 영역 내부에서 렌더링되면 부모의 스크롤 컨테이너에 갇혀버려서, `Retention.jsx`에서 `createPortal(..., document.body)`로 `document.body`에 직접 띄우도록 고쳤다. 다이얼로그를 새로 만들 때도 (App.jsx 최상위에서 렌더링하지 않는 이상) 이 패턴을 따라야 같은 문제가 재발하지 않는다.
- **한글 PDF**: `jsPDF` 기본 폰트는 한글 글리프가 없어서 `pdfExport.js`가 `html2canvas`로 화면을 이미지 캡처한 뒤 PDF에 삽입하는 방식으로 우회했다. 텍스트 선택은 안 되는 이미지 기반 PDF다 — 텍스트 선택이 필요해지면 한글 폰트(예: `app/public/ds/assets/fonts/` 안의 SamsungOneKorean)를 jsPDF에 `addFont`로 임베드하는 작업이 별도로 필요하다.
- **고객사 제출 자료 원칙**: 생애가치(BEP/ROI·할인율 등)는 리포트 팝업(`MonthlyReportDialog`)에 절대 넣지 않는다. 내부용 정보는 고객 상세의 `LifetimeValueBox`에서만 보인다. 리포트에 새 섹션을 추가할 때 이 경계를 유지해야 한다.
- 번들 크기: `jspdf`+`html2canvas` 추가로 메인 번들이 커졌다(빌드 시 500KB 초과 경고). 지금은 문제없지만, 나중에 느려지면 `import()` 동적 로딩으로 PDF 관련 코드만 분리하는 걸 고려한다.

## 6. 확인이 더 필요한 부분
- PDF 다운로드 버튼의 실제 캡처·저장 동작은 이 환경에 headless 브라우저가 없어 직접 클릭 테스트를 못 했다. 실제 브라우저에서 한 번 확인 필요.
- 월간 리포트 발송("승인 후 발송 처리")과 감성터칭 메시지 발송("승인 발송(모의)")은 둘 다 날짜만 화면에 반영하는 모의 동작이다 — 실제 이메일/카카오/문자 발송 연동은 후속 과제(R23·R25 완성 범위 밖).
- 감성터칭 메시지의 발송 타이밍(정확히 며칠 뒤가 적절한지)은 사람이 가이드라인을 정해야 하는 영역(PRD: "톤·발송 타이밍 가이드라인 설정"은 사람 몫) — 지금은 버튼을 아무 때나 누를 수 있고 타이밍 강제 로직은 없다.
