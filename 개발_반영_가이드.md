# 개발 반영 가이드 (비개발자용)

## 0. 먼저 확인할 것 — 이 폴더는 아직 깃(Git) 저장소가 아님

로컬에 `.git`이 없다. 즉 지금까지의 작업은 깃허브 어디에도 연결되어 있지 않다.
개발자에게 전달할 때 아래 둘 중 하나를 먼저 정해야 한다.

- 팀 깃허브에 **기존 레포**가 있다면 → 그 레포를 clone 받아서, 이 폴더의 `app/` 내용물을 그 레포 안에 덮어써 달라고 요청
- 아직 레포가 없다면 → 이 `app/` 폴더를 새 레포의 시작점으로 삼아 최초 커밋해 달라고 요청

## 2. 깃허브에 올릴 것 / 올리지 않는 게 나은 것

**올릴 것 (실제 앱 코드)**
`app/` 폴더 전체 — `src/`, `public/`(데이터·디자인시스템 포함), `scripts/`, `package.json`, `vite.config.js`, `README.md`

**기획 문서** — 팀 정책에 따라 `app/`과 같은 레포의 `docs/` 폴더에 두거나, 별도 문서 저장소(Notion 등)에 두면 됨
`PRD_v3_수정본.md`, `BUILD_SPEC.md`

**보통 올리지 않는 것**
- 최상위 HTML 목업 3개, `Bluescan loading spinner design.zip` — 실행되는 앱이 아니라 디자인 참고 자료(파일당 4~6.5MB로 용량도 큼). 올리려면 레포 대신 `docs/mockups/` 같은 별도 폴더에 두거나 압축 안 한 채로 공유 링크로 전달 추천.
- `app/.env` (API 키) — 이미 `.gitignore`에 포함되어 안전하게 제외됨. `.env.example`만 레포에 남기고, 실제 키 값은 절대 깃허브에 올리면 안 됨.

## 3. 개발자에게 전달할 때 체크리스트
1. 이 폴더 전체(특히 `app/`) 전달 또는 접근 권한 공유
2. "테마 A 잔여 기능(R16~20)"과 "테마 B 신규 개발(R21~26)" 중 우선순위 지정 — PRD 참고
3. 참고 디자인 목업(`블루스캔_거절기능추가_PRD정합보정.html`) 위치 안내
4. API 키 관련 — `.env.example` 참고해 개발자가 개인 키로 로컬 설정하도록 안내(키를 대신 전달하지 말 것)

## 4. 화재 출동 통계 실연동 — 담당자(김두환)에게 전달할 때
직접 API 연동을 하지 않고 담당자에게 넘기는 경우, 아래 3가지만 함께 전달하면 된다.
1. **지금 상태**: `app/src/fireDispatch.demo.js` — 화면에 필요한 데이터 구조(시도별 접수/진행/인명/재산피해)를 이미 목업으로 구현·검증해둔 파일. "이 구조 그대로 실데이터로 채워달라"고 요청하면 됨.
2. **데이터 소스 참고자료**: `SKILL.md`(daily-fire-dashboard-briefing) — 소방청 오픈API 엔드포인트, 파라미터, safekorea.go.kr 실시간 수집 방식이 이미 정리돼 있어 처음부터 조사할 필요가 없음.
3. **연결 지점**: `app/src/screens/Dash.jsx`에서 `buildFireDispatchDemo()`를 부르는 부분 — 실제 데이터 함수로 이름만 바꿔 끼우면 화면은 그대로 재사용됨.

담당자가 정할 것: 실시간으로 얼마나 자주 갱신할지(매일 1회로 충분한지), 서버에서 직접 호출할지 별도 배치 잡을 둘지. 이건 기술적 판단이 필요한 부분이라 서지나님이 미리 결정할 필요는 없다.
