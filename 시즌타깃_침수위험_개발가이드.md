# 시즌 타깃(침수 위험) 개발 가이드

엑셀 고도화 항목 "시즌별 특화 타깃"(담당 서지나)의 1차 구현. 지금은 전부 **목업/기존 실데이터 재사용 혼합** 상태이고,
이 문서는 (1) 지금까지 만든 파일 구조, (2) 그중 뭐가 진짜고 뭐가 가짜(목업)인지, (3) **원본 더미데이터를 실데이터로 바꾸는 방법**을 정리한다.

## 1. 지금 구현된 것 요약

| 기능 | 실제 데이터인가? | 근거 |
| --- | --- | --- |
| 리스트 A 건물의 침수 위험(`c.flood`) | **실제** | `build-data.mjs`가 빌드 시점에 진짜 침수 GeoJSON(도시침수 기왕최대)과 건물 좌표를 point-in-polygon 대조해서 계산 |
| 리스트 B 고객처의 침수 위험 | **실제 계산 + 좌표 정확도는 목업 수준** | `floodRisk.js`가 리스트 A와 같은 방식(같은 GeoJSON)으로 클라이언트에서 계산. 판정 로직은 진짜지만, 리스트 B 좌표 자체가 원본 데이터에 정밀 지오코딩이 안 되어 있어서 대부분 폴리곤 밖으로 잡힘 |
| 서강지사(마포) 시연용 좌표 보정 | **목업(하드코딩)** | `floodRisk.js`의 `DEMO_RELOCATE` — 고객 2곳 좌표를 실제 마포 침수폴리곤 내부로 강제 이동 |
| 지하 중요시설 세부 태그(지하 전기실 등) | **완전 목업** | `floodRisk.js`의 `demoUndergroundRoomFor()` — id 해시로 6개 태그 중 하나를 결정론적으로 배정. 원본 데이터에 이 필드 자체가 없음 |
| 시즌 게이팅(6~9월만 표시) | **실제 로직** | `season.js`의 `currentSeasonKey()` — 실제 날짜 기준. 대시보드에서 미리보기로 강제 전환 가능(시연용 기능이지 목업 데이터 아님) |

## 3. 원본 더미데이터를 실데이터로 바꾸는 방법 (핵심)

### 3-1. 리스트 B 고객처 좌표 정밀도 — 가장 먼저 해결해야 할 것

지금 리스트 B의 `lat`/`lng`는 `build-data.mjs`가 지사·주소 기반으로 대략 배치한 값이라, 실제 건물 위치와 오차가 크다.
이 오차 때문에 실제로는 침수구역에 있는 고객처인데도 폴리곤 밖으로 잡히는 경우가 많다(922건 중 정밀 좌표 없이는 6건만 잡힘).

**해야 할 일**: 리스트 B 원본 CSV(`upsell_manned_guard_status.csv`, `upsell_critical_rooms.csv`)에 있는 주소를
도로명주소 API 또는 카카오/네이버 지오코딩 API로 정밀 위경도 변환한 뒤, `build-data.mjs`의 리스트 B 생성 구간
(`loadUpsell(...)` 호출부, 약 441번째 줄 부근)에서 이 정밀 좌표를 `lat`/`lng`로 채워 넣는다.

정밀 좌표가 들어오면 `floodRisk.js`는 코드 수정 없이 그대로 더 정확하게 동작한다(판정 로직은 이미 실제 지오메트리 기준이라 손댈 필요 없음).

### 3-2. 서강지사 시연용 좌표 보정(`DEMO_RELOCATE`) 제거

`app/src/floodRisk.js`에 있는 아래 블록은 3-1이 끝나 실제 좌표가 정확해지면 **삭제**한다(그때는 실제 좌표만으로도
서강지사 고객 중 침수구역 안에 있는 곳이 자연스럽게 잡혀야 정상):

```js
const DEMO_RELOCATE = {
  B2318: [126.906988, 37.5727],
  B2512: [126.877507, 37.58614],
};
```

`augmentListBFlood()` 안에서 `DEMO_RELOCATE`를 참조하는 `relo` 관련 3줄도 함께 지우고, `floodRiskOf(b.lat, b.lng)`를
바로 호출하도록 되돌리면 된다.

### 3-3. 지하 중요시설 태그 — 실데이터로 교체

가장 손이 많이 가는 부분. 현재 원본 데이터(중요실 CSV)에는 "전산실·기계실" 같은 뭉뚱그린 값만 있고,
"지하 여부"·"세부 설비 종류"가 없어서 완전히 목업(`demoUndergroundRoomFor`)으로 채워뒀다.

**필요한 원본 데이터 보강**: `upsell_critical_rooms.csv`에 아래와 같은 컬럼을 추가로 조사·기입해야 한다.

| 신규 컬럼 예시 | 값 예시 |
| --- | --- |
| `is_basement` | Y / N |
| `room_type_detail` | 전기실(수배전반) / 발전기실 / 펌프실(배수펌프) / 기계실 / 통신실(MDF) / 서고·문서고 |

**반영할 코드 위치**:
1. `app/scripts/build-data.mjs`의 `keywordForRoom(ind)` 함수(약 406번째 줄) — 지금은 업종명으로 room을 "추정"하고 있다. CSV에 실제 컬럼이 생기면 이 추정 로직 대신 CSV 값을 그대로 읽어와 `room`(기존 필드) 또는 신규 필드 `undergroundRoom`으로 채운다.
2. `app/src/floodRisk.js`의 `demoUndergroundRoomFor()` 호출부(`augmentListBFlood()` 안) — id 해시로 태그를 지어내는 대신, `b.undergroundRoom`(위 1번에서 CSV로부터 채워진 실제값)이 있으면 그걸 쓰고 없으면 `null`(=지하 중요시설 정보 없음, NO_DATA 취급)로 바꾼다. `UNDERGROUND_CRITICAL_ROOMS` 배열은 CSV 값의 표준 라벨 목록으로 남겨서 검증(밸리데이션)용으로 재사용 가능.
3. 화면(`ListB.jsx`, `Dash.jsx`)은 `c.undergroundRoom` 필드를 그대로 읽고 있으므로 **화면 코드는 수정할 필요 없음** — 데이터만 실제 값으로 바뀌면 자동 반영된다.

### 3-4. 리스트 A 침수 커버리지 확장 (선택 — 후속)

지금 침수 GeoJSON은 12개 시군구(서울 11개 구 + 경기 일산서구)만 있다. 전국 확장은 `app/scripts/convert-flood.mjs`가
원본 침수 shapefile을 GeoJSON으로 변환하는 스크립트이므로, 추가 지역 shapefile을 구해 이 스크립트를 다시 돌리고
`app/public/flood/manifest.json`에 항목을 추가하면 된다. 이후 `npm run build`(또는 `node scripts/build-data.mjs`)를
다시 실행하면 리스트 A·B 모두 확장된 커버리지로 자동 반영된다(리스트 B는 클라이언트에서 매번 새로 계산하므로
재빌드만 하면 되고, 별도 코드 수정 불필요).

## 4. 시즌 확장(후속 — 동절기 등)

`app/src/season.js`에 `winter`(동절기, 전산실·냉동창고 타깃) 자리는 이미 만들어뒀지만 실제 타깃 로직은 없다.
같은 패턴으로 확장하려면:
1. 동절기에 위험한 타깃 조건 정의(예: 냉동창고·수영장 등 업종 키워드) → `listA`/`listB`에 이미 있는 `use`/`ind` 필드로 필터링 가능한지 우선 검토
2. `floodRisk.js`와 같은 형태의 `winterRisk.js` 신규 파일로 판정 로직 작성
3. `Dash.jsx`의 "시즌 타깃 신호" 카드에 `seasonKey === 'winter'`일 때 보여줄 분기 추가(지금은 침수(flood)만 있고 winter는 "시즌 아님" 안내만 뜸)
