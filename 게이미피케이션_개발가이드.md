# 게이미피케이션 개발 가이드

엑셀 고도화 로드맵의 "B.T.S Quest"(방문·컨설팅키트·쇼츠제작·수주·감성터칭 5단계 퀘스트 + 실시간 랭킹 + 기프티콘 보상, 서지나님 담당) 아이디어를 이 앱의 실제 화면 기준으로 구체화한 1차 구현이다. 목적은 방금 만든 유지고객 리포트·감성터칭 같은 신규 기능의 사용률을 끌어올리는 것.

## 1. 설계 원칙 — "파생 뷰"

게이미피케이션 전용 데이터베이스나 이벤트 로그를 새로 만들지 않았다. 앱에 이미 있는 상태(`visits`, 유지고객 리포트/감성터칭 발송 여부)를 그대로 읽어서 포인트·퀘스트 진행률·배지를 **매번 다시 계산**한다. 장점은 이중 기록 걱정이 없다는 것, 단점은 세션이 끝나면(새로고침·재로그인) 포인트가 초기화된다는 것 — 실서비스 전환 시 반드시 영구 저장이 필요하다(4절 참고).

## 2. 파일 목록

| 구분 | 파일 | 내용 |
| --- | --- | --- |
| 신규 | `app/src/gamification.js` | 포인트 계산·퀘스트·배지·리더보드 — 순수 함수만 있음(화면 코드 의존 없음) |
| 신규 | `app/src/toastMessages.js` | 결과 저장 시 뜨는 응원 팝업 문구·아이콘 매핑 |
| 수정 | `app/src/App.jsx` | 포인트 계산(`gamify`), 상단바 포인트 배지, 토스트 큐·트리거, 유지고객 리포트/감성터칭 발송 상태를 `Retention.jsx`에서 끌어올림 |
| 수정 | `app/src/screens/Dash.jsx` | 컨설턴트 대시보드에 "이번 주 퀘스트·랭킹" 섹션 추가 |
| 수정 | `app/src/screens/Retention.jsx` | 리포트/감성터칭 발송 상태를 로컬 state 대신 App.jsx props로 받도록 변경 |
| 수정 | `app/src/app.css` | `.gm-badge`(배지 칩), `.gm-toast*`(응원 팝업), `.topbar__points*`(상단바 포인트) |

## 3. 동작 구조

```
App.jsx
  ├─ visits, retention, reportSentOverrides, touchOverrides (기존/기존에 끌어올린 state)
  ├─ gamify = computeSessionPoints(ctx)         → 상단바 포인트 배지, "+N" 반짝임
  ├─ quests = questProgress(ctx), badges = badgeStatus(ctx)
  │     └─ 이전 상태와 diff → 새로 달성/획득된 것만 toastMessages로 팝업(gm-toast)
  └─ SalesDash(Dash.jsx)에 gamify/quests 원본 ctx를 넘겨 "이번 주 퀘스트·랭킹" 섹션 렌더
```

### 3-1. 포인트 (`gamification.js`의 `POINTS`)
| 액션 | 배점 |
| --- | --- |
| 방문완료 기록 | 10 |
| 재방문필요 기록 | 5 |
| 거절(사유 기록) | 5 |
| 수주완료 | 50 |
| 화재·침수 등 타이밍 신호 후보 방문(보너스) | +10 |
| 유지고객 월간 리포트 발송 | 15 |
| 유지고객 감성터칭 메시지 발송 | 10 |
| 위 두 액션을 "주의 필요" 고객 대상으로 하면(보너스) | +10 |

**이 숫자는 전부 예시다.** 실제 배점·난이도는 영업 관리자가 검토·확정해야 하는 영역(원안의 "사람이 할 것"과 동일한 원칙) — 코드에서 임의로 자동 조정하지 않는다.

### 3-2. 퀘스트 (`QUEST_DEFS`)와 배지(`BADGE_DEFS`)
현재 4개 주간 퀘스트(방문 5건, 타이밍 신호 대응 3건, 감성터칭 2건, 리포트 1건)와 6개 배지(첫 방문, 첫 계약, 안전 지킴이, 리텐션 수호자, 완벽한 기록가, 퀘스트 마스터)가 있다. 전부 `ctx`(visits/listA/listB/retention/reportSentOverrides/touchOverrides)만 보고 계산하는 순수 함수라, 새 퀘스트·배지를 추가할 땐 이 두 배열에 항목만 더하면 된다.

### 3-3. 리더보드 (`buildLeaderboard`)
로그인한 본인은 **실제 세션 포인트**, 같은 지사의 다른 컨설턴트는 기존 `Dash.jsx`의 `consultantPerf`와 같은 방식(사번을 시드로 한 결정론적 난수)의 **예시 점수**를 섞어서 순위를 매긴다. 즉 지금은 "나만 진짜로 움직이고 동료는 고정된 가짜 숫자"인 상태 — 실서비스에서는 동료 데이터도 실제 활동 기반으로 바뀌어야 한다(4절).

### 3-4. 응원 팝업 (`toastMessages.js`, App.jsx)
포인트가 오르거나 배지·퀘스트가 새로 달성되면 화면 우측 하단에 팝업이 뜬다. 상황별(방문/수주/타이밍대응/리포트/감성터칭/배지/퀘스트)로 문구 풀이 따로 있고 매번 랜덤하게 하나를 고른다 — 문구·톤을 바꾸고 싶으면 이 파일의 `POOLS` 배열만 수정하면 된다.

**주의**: 앱이 처음 뜰 때 이미 깔려있는 데모 시드(방문 3건 등)에는 팝업이 뜨면 안 돼서, `App.jsx`의 각 diff용 `useRef`를 `null`로 시작해 "첫 실행에는 기준값만 저장하고 팝업은 안 띄운다"는 가드를 넣어뒀다. 비슷한 diff 로직을 추가할 때 이 가드를 빼먹으면 마운트 직후 팝업이 우르르 뜨는 버그가 생긴다.

## 4. 실서비스 전환 시 할 일

### 4-1. 포인트 영구 저장
지금은 세션(새로고침 전까지)에 한정된 파생 계산이다. 실전환 시:
- 서버에 사용자별 포인트 원장(`points_ledger` 같은 테이블)을 두고, 방문 저장·리포트 발송·감성터칭 발송 등 각 액션이 일어날 때 서버에 적립 이벤트를 기록
- `computeSessionPoints`의 배점 로직(`POINTS` 상수)은 그대로 서버 쪽으로 옮기면 된다 — 계산식 자체는 이미 실제 로직

### 4-2. 동료 데이터 실데이터화
`buildLeaderboard`의 `seededPoints()`(가짜 점수 생성기) 호출을 제거하고, 서버에서 같은 지사 컨설턴트들의 실제 누적 포인트를 조회해 넣으면 된다. 함수 시그니처(`consultants, myEmpno, myPoints`)는 그대로 두고 `myPoints` 자리에 서버 값을 넣게 바꾸면 화면 코드는 안 건드려도 된다.

### 4-3. 퀘스트 "이번 주" 범위
지금은 "이번 세션 동안의 활동"을 퀘스트 진행률로 쓴다(실제 주간 단위가 아님). 실전환 시 `QUEST_DEFS`의 `current()` 함수들이 날짜 범위(이번 주 월요일~오늘)로 필터링하도록 고쳐야 한다.

### 4-4. 보상(기프티콘 등)
원안대로 AI는 트리거(배지/퀘스트 달성 감지)까지만 하고, 실제 보상 지급·예산·운영 정책은 사람이 설계해야 하는 영역이다. 지금은 보상 지급 기능 자체가 없다(배지·퀘스트 달성 알림까지만).

## 5. 확인이 더 필요한 부분
- 토스트 팝업·상단바 애니메이션은 빌드까지만 확인했고, 실제 브라우저에서 여러 액션을 빠르게 연속으로 했을 때 팝업이 자연스럽게 쌓이고 사라지는지는 직접 확인이 필요하다(headless 브라우저 없음).
- 포인트/퀘스트/배지 계산 로직 자체는 Node로 실제 데이터를 넣어 검증 완료(합산·진행률·잠금해제 정상 동작).
