# 시니어 UI/UX 학습 정리 + 블루스캔 적용안

## 한 줄 요약

시니어 UX의 핵심은 "노인 전용 앱"을 따로 만드는 게 아니라, **가장 취약한 사용자를 기준으로 설계하면 모두가 편해진다(유니버설 디자인)**는 것이다. 시니어는 시각·인지·운동 능력의 변화 때문에 (1) 무엇이 눌리는지 모르고, (2) 예시를 자기 정보로 착각하고, (3) 스크롤을 안 하고, (4) 질문형 문구를 헷갈려 하고, (5) 단계가 깊으면 시작조차 포기한다. 블루스캔은 현재 **PC 중심의 데이터 밀집 대시보드(본문 14px, 캡션 12~13px, 파란색 단일 신호 의존)**라, 이 원칙들을 적용하면 시니어뿐 아니라 현장 영업담당·관리자 전반의 사용성이 함께 올라간다.

# Part 1. 시니어 UI/UX 학습 정리

## 1. 시니어는 한 덩어리가 아니다 — 디지털 리터러시 3유형

상명대 논문(FGI 10명)과 한국보건사회연구원 분류에 따르면 시니어(55세+)는 디지털 역량에 따라 세 부류로 나뉜다. **설계 기준선은 항상 가장 낮은 집단('디지털 소외형')에 맞춘다.**

| 유형 | 비중(참고) | 특징 | 설계 시사점 |
| --- | --- | --- | --- |
| 디지털 소외형 | 약 24.7% | 두려움·거부감으로 시도조차 안 함. 기기·장소마다 인터페이스가 달라 혼란 | **최소 단계·핵심 기능만.** IA Depth 1~2단계. 이 집단을 기준으로 설계 |
| 저역량·저활용형 | 약 36.2% | 접근은 되나 자유롭게 못 씀. 조력자 필요 | 익숙한 패턴·명확한 위치 안내로 자립 지원 |
| 적극 활용형(액티브 시니어) | 약 39.1% | 젊은 층과 유사. 콘텐츠 양·질까지 요구 | 부가 기능·심화 정보 제공, 단 기본은 단순하게 |

> 핵심: UI 선호(큰 글씨·높은 대비·직관성)는 **모든 시니어 공통**이고, UX 선호(기능 깊이·부가 기능)만 역량별로 갈린다. 그래서 **UI는 가장 취약한 기준으로 통일하고, 심화 기능은 선택적으로 얹는다.**

## 2. 노화에 따른 3대 변화와 설계 함의

### (1) 시각 — 만 40세부터 노안, 50세 이상 절반이 시력 저하

- **글씨**: 본문 16pt 이상 권장. 사용자가 크기를 조절할 수 있는 옵션 제공. 서체는 1~2종으로 통일, 장식적 폰트 대신 익숙한 산세리프(예: Noto Sans KR / Pretendard).
- **대비**: 텍스트·이미지 최소 **4.5:1**, 큰 글씨는 최소 3:1. **그레이스케일로 바꿔도 다 읽히는지** 확인.
- **색**: 색채 대비 감도가 떨어지고 **파란색 음영이 특히 희미하게 보인다.** 중요한 아이콘·신호를 파란색 계열 단독으로 쓰지 말 것. 적록색맹 고려해 빨강/초록만으로 의미 전달 금지 → 색 대신 **굵기·아이콘·라벨 병행**.
- **정보량**: 다 보여주기보다 **찾는 정보만 확실히 강조**(네이버 지도 vs 카카오 지도 예시). 이미지 위에 텍스트 겹치지 않기.

### (2) 인지 — 기억·주의력·의사결정

- **기억(단기·삽화적 기억 약화)**: 새 기능 학습에 시간이 오래 걸림 → **One thing per page**(한 화면 한 기능). 진행 완료에 대한 **명확한 실시간 피드백**(마이크로 인터랙션). 습관적 행동은 미리 알림으로 유도.
- **주의력(멀티태스킹 저하, 단 지속적 주의력은 오히려 높음)**: 화면 **중앙에 핵심 배치**. 상호작용 **시간 제한을 넉넉히**. 긴 텍스트를 두려워 말 것 — 시니어는 호흡 긴 글에 오히려 익숙하다.
- **의사결정**: **이미 아는 지식·익숙한 패턴**으로 결정. 새 디자인보다 **친숙한 레이아웃·색상**. **권위 있는 전문가 의견·근거**를 신뢰의 앵커로 노출.

### (3) 운동 제어 — 손 떨림(수전증), 정밀 조작 저하

- **터치 타깃**: 최소 44×44px(iOS 기준) 이상. 아이콘이 작거나 복잡하지 않게.
- **입력 최소화**: 과도한 타이핑·탭 지양. **초성 검색·자동완성·선택형 입력**으로 입력량을 줄인다.

## 3. 토스가 관찰한 시니어 5대 오인지 패턴 (실전 체크리스트)

가장 실무적으로 유용한 부분. 730만 시니어 사용자 UT에서 반복 관찰됨.

1. **컴포넌트 오인지** — 텍스트·화살표(>)가 눌리는 줄 모른다. 태그와 버튼을 구분 못 해 엉뚱한 곳을 누른다. → **진입점은 명확한 버튼 형태**로. 리스트에 진입점 종류가 여러 개면 특히 위험.
2. **예시 이미지 오인지** — 예시 화면(특히 이름·금액 등 개인화된 데이터가 움직이면)을 **자기 실제 정보로 착각**한다. → 예시엔 크고 분명한 '예시' 표식, 개인화 텍스트 사용 주의.
3. **모션 그래픽 오인지** — 안내 문구보다 캐릭터 모션에 집중해 **동작을 그대로 외워 따라 하려다 실패**한다. → 실제 사용자가 할 행동을 문구로 명확히.
4. **라이팅 오인지** — "어떤 서류가 필요한가요?" 같은 질문형을 **'나에게 질문한다'로 오해**한다. → **"필요한 서류 확인하기"** 같은 행동 지시형으로.
5. **화면 구조 오인지** — 스크롤 단서가 없으면 **아예 스크롤하지 않고** 보이는 것만으로 판단한다. → 스크롤 가능함을 시각적으로 암시(잘린 콘텐츠 노출 등).

## 4. UI/UX 설계 원칙 종합 (논문 기준 표 정리)

**UI (모든 시니어 공통)**

- 레이아웃: 하단 스크롤형 **싱글 페이지**, 심플·정돈, 기능/메뉴 직관 배치.
- 색채: 밝고 화사한 톤 + 높은 대비. (단, B2B 브랜드라면 대비·명료성 원칙만 취하고 파스텔 강제는 지양)
- 타이포: 산세리프 + 큰 크기 + 배경 대비되는 선명한 색.
- 아이콘: **크게 + 그림과 텍스트 동시 표기**(아이콘 단독 금지), 스타일 일관성.

**UX (역량별 차등, 기준은 소외형)**

- 조작적: **IA Depth 1~2단계**, "지금 어디에 있는지" 위치 명시, 홈 버튼 등 핵심 기능 쉽게 발견.
- 인지적: **한국어 용어**(외국어 취약), 아이콘+텍스트 결합으로 가독성.
- 반응적: 행동 즉시 피드백(약한 주의력 보완).
- 행동 유도: 화면 전환 등 변화를 **시각적으로 두드러지게**, 인터랙션 자연스럽게 유도.

## 5. 가장 중요한 통찰

- **"노인이 가장 싫어하는 디자인은 노인 전용 디자인이다."** 선 긋지 말 것. 일반 앱에 시니어 원칙을 녹여라.
- **유니버설 디자인의 원리**: 지금 불편한 사람(시니어)의 문제를 풀면 잠재적 문제를 겪는 더 많은 사람(2040 초보자, 급한 상황의 사용자 등)까지 해결된다.
- 토스 사례의 반론(댓글)도 새겨둘 것: **"이건 시니어만의 문제가 아니라 기획·디자인 단계의 결함"**이라는 지적. 즉 시니어 기준 개선 = 전체 사용성 개선.

# Part 2. 블루스캔 타깃 에이전트 적용안

## 0. 현재 상태 진단

블루스캔은 컨설턴트·영업/CS 관리자용 **PC 중심 데이터 밀집 웹앱**이다(에스원 S-1 디자인시스템 기반). 디자인 토큰·화면을 확인한 결과, 시니어 관점에서 다음이 걸린다.

| 관찰된 현재 상태 | 시니어 관점 리스크 |
| --- | --- |
| 본문 기본 14px, 캡션 12~13px, 마이크로 10px 사용 | 노안 사용자에게 캡션·수치·라벨이 안 읽힘 |
| 파란색(`--s1-blue-500`)을 핵심 신호/강조에 단독 사용 | 파란색 음영은 시니어가 가장 못 보는 색. 신호 놓침 |
| gray-500/400 텍스트(#757575, #9d9d9d)로 보조 정보 표기 | 흰 배경 대비 4.5:1 미달 가능 → 저대비 문제 |
| 대시보드에 리스트A/B·화재신호·방문결과·유지대시보드가 동시 밀집 | 주의 분산, "어디를 눌러야 하는지" 혼란 |
| 화살표(>)·태그·칩 등 다양한 진입점 혼재 | 무엇이 눌리는지 오인지 (토스 패턴 1) |
| 데모/예시 데이터에 실제처럼 보이는 사업장명·수치 | 예시를 실제로 오인 (토스 패턴 2) |
| 아이콘 폰트(Material Symbols) 의존 | 로드 실패 시 텍스트("expand_more") 노출, 아이콘 단독은 의미 불명 |

> 참고: 블루스캔의 1차 사용자는 현재 시니어가 아닐 수 있으나, **현장 영업담당 중 중장년 비중·관리자 연령대**를 고려하면 시니어 원칙 적용은 곧 전 사용자 사용성·시연 안정성 향상으로 이어진다.

## 1. 즉시 반영 가능한 토큰·전역 수준 개선 (효과 대비 저비용)

1. **본문 기준 상향**: body 14px → **16px**, 캡션 12px → **14px**, 10px 마이크로는 화면 표기에서 제거하거나 12px로. 수치·D-day·점수 등 핵심 데이터는 16~18px.
2. **텍스트 대비 강화**: 보조 텍스트를 gray-400/500 → **gray-700 이상(#434343~)**으로. 흰 배경 기준 4.5:1 충족 검증(그레이스케일 테스트).
3. **색 단독 신호 금지**: 위험/정상/주의를 **파란색·빨강 색상만으로** 구분하지 말고 **아이콘 + 텍스트 라벨 + 굵기**를 함께. (예: 이탈위험 "높음"을 빨강 점만이 아니라 `⚠ 이탈위험 높음`으로)
4. **글자 크기 조절 옵션** 제공(최소 '보통/크게' 토글).
5. **아이콘은 항상 텍스트 라벨 동반**, 아이콘 폰트 로드 실패 대비 fallback 텍스트/SVG.

## 2. 화면별 적용 체크리스트

### 대시보드 (Dash)
- [ ] "실시간 화재 출동" 등 **핵심 지표를 화면 중앙·상단에 크게**, 부가 위젯은 아래로.
- [ ] 위젯이 많아 **주의 분산** → 카드별 카테고리·제목을 명확히 구분(One thing per card).
- [ ] 스크롤 하단에 콘텐츠가 있으면 **잘린 카드 일부를 노출**해 스크롤 단서 제공(토스 패턴 5).
- [ ] 날짜 라벨의 "오늘/실시간"과 실제 데이터 날짜 불일치 제거 → **신뢰(의사결정 앵커)** 훼손 방지.

### 방문 (Visit)
- [ ] "AI로 결과 분석" 버튼 → **행동 지시형 라벨** 유지하되 처리 상태(로딩·완료)를 **즉각적·명확한 피드백**으로.
- [ ] 분석 결과가 **규칙 기반인지 LLM인지** 등 근거·출처를 명시(시니어는 전문가·근거 신뢰 중시).
- [ ] 입력 필드는 **선택형·자동완성** 우선, 자유 타이핑 최소화(운동 제어).
- [ ] 저장 완료 시 "무엇이 어디에 저장됐는지" 명확한 확인 메시지.

### 리스트 A / B (ListA, ListB)
- [ ] 행(row) 진입점을 **명확한 버튼/카드 형태**로. 화살표(>)·태그 단독 진입 지양(토스 패턴 1).
- [ ] 점수·NO_DATA·B-1/B-2 유형 등 **약어·기호를 한국어 라벨로 풀어** 표기(인지적 요소).
- [ ] 한 행에 정보가 많으면 **핵심(사업장명·점수·다음 액션)만 강조**, 나머지는 상세로.

### 로그인 (Login)
- [ ] 필드·버튼 **터치 타깃 44px+**, 큰 라벨.
- [ ] 오류 메시지는 색상만이 아니라 **아이콘+문장**으로 원인·해결 안내.

### 예시/데모 데이터 (retention.demo, fireDispatch.demo 등)
- [ ] 예시 화면·샘플 대사에 **크고 분명한 '예시/데모' 배지** — 실제 데이터 오인 방지(토스 패턴 2).

## 3. 우선순위 (효과 × 난이도)

| 우선순위 | 항목 | 근거 원칙 | 난이도 |
| --- | --- | --- | --- |
| P0 | 본문/캡션 폰트 상향 + 텍스트 대비 강화 | 시각(노안·대비) | 낮음(토큰) |
| P0 | 색 단독 신호 → 아이콘+라벨 병행 | 시각(파란색·색맹) | 낮음~중 |
| P1 | 진입점 버튼화(화살표/태그 지양) | 토스 패턴 1 | 중 |
| P1 | 예시/데모 배지 명시 | 토스 패턴 2 | 낮음 |
| P1 | 질문형→행동지시형 라이팅 점검 | 토스 패턴 4 | 낮음 |
| P2 | 대시보드 중앙 집중·스크롤 단서 | 주의력·패턴 5 | 중 |
| P2 | 입력 최소화(선택형·자동완성) | 운동 제어 | 중~높음 |
| P2 | 글자 크기 조절 옵션 | 시각 | 중 |

## 4. 다음 단계 제안

1. 위 P0 항목부터 디자인 토큰(`typography.css`, `colors.css`) 수정 → 전 화면 일괄 반영.
2. 시니어(또는 중장년 영업담당) 3~5명 대상 **간이 UT**: "이 화면에서 뭘 눌러야 할까요?"를 소리 내 말하게 하고 오인지 지점 수집(토스 방식).
3. 오인지 패턴을 팀 가이드라인으로 문서화해 신규 화면에 재사용.
