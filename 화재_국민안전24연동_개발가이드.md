# 화재 스코어링 · 국민안전24 연동 개발 가이드

지금 앱(`app/src/fireDispatch.demo.js`)에는 두 축의 목업이 붙어있다.

1. **전국 요약 배지** — 접수/진행/인명/재산피해 (`national`, `bySido`에서 합산). 시도별 막대 나열은 영업 관점에서
   의미가 약해 화면에서 뺐다(데이터 자체는 `bySido`에 남아있음 — 다른 화면에서 필요해지면 재사용 가능).
2. **실시간 화재 출동 리스트(`liveItems`)** — 컨설턴트 대시보드에 뉴스처럼 스크롤되는 개별 출동 건 목록으로,
   그리고 리스트 A·B 지도에는 🚒 마커 오버레이로 표시된다(`app/src/map.jsx`, `ListA.jsx`, `ListB.jsx`도 이제
   이 데이터를 함께 사용한다 — 아래 표 참고).

이 목업을 실제 데이터로 만드는 방법은 두 가지다. 아래 중 하나를 골라 개발자에게 그대로 전달하면 된다.

- **방법 A** — 계속 목업으로 두고 정교화만 한다 (실시간 연동 없음)
- **방법 B** — 서지나님이 이미 돌리고 있는 `daily-fire-dashboard-briefing` 스케줄(매일 아침 국민안전24·소방청 데이터 수집)을 재사용해 블루스캔 앱에 실제로 흘려보낸다

---

## 방법 A. 목업 데이터 고도화 (실연동 없음)

실시간일 필요가 없다면 이 방법으로 충분하다. 개발자에게 "이 파일 구조 그대로 숫자만 더 현실감 있게 채워달라" 정도로 전달 가능.

| 작업 | 파일 | 내용 |
| --- | --- | --- |
| 수정 | `app/src/fireDispatch.demo.js` | `BY_SIDO`·`LIVE_ITEMS` 배열 확장(시도 수↑, 출동 건수↑, 다양한 지역 추가 등). 반환 구조(`national`/`bySido`/`liveItems`/`baseDate`/`source`)는 유지 |
| 수정 없음 | `app/src/screens/Dash.jsx`, `map.jsx`, `screens/ListA.jsx`, `screens/ListB.jsx` | 이미 `buildFireDispatchDemo()`로 연결되어 있어 추가 작업 불필요 |
| 추가/삭제 없음 | — | 새 파일도, 지울 파일도 없음 |

장점: 개발 리소스 거의 안 듦, 브라우저 호출·API 키·스케줄 인프라 불필요.
단점: 화면에 뜨는 숫자가 실제 오늘자 화재 현황이 아님(예시 데이터라는 라벨은 이미 화면에 표시 중).

## 방법 B. 스케줄 실연동 (매일 자동 갱신)

서지나님의 기존 스케줄(`daily-fire-dashboard-briefing`, 매일 아침 소방청 오픈API + safekorea.go.kr 수집 → `midoseo/fire_dashboard` 깃허브 → Netlify 자동배포)을 그대로 재사용한다. 이미 매일 도는 파이프라인이 있으므로 **새로 브라우저 자동화를 만들 필요는 없고, 그 결과물을 블루스캔 앱이 읽어가게만 하면 된다.**

> 주의: `SKILL.md`(스케줄 지침 파일)는 Cowork Settings에 등록된 실제 스케줄 스킬이라 이 세션에서 제가 직접 수정할 수는 없다. 아래 변경 내용은 서지나님이 Settings > Capabilities에서 스케줄을 열어 직접 반영하거나, 스케줄 관리 담당자에게 전달해야 한다.

### 구조
```
[매일 아침 스케줄] 국민안전24·소방청 API 수집
      │
      ├─ 1단계 결과(시도별 통계) ─┐
      ├─ 2단계 결과(safekorea 실시간 화재정보, 전 페이지) ─┤
      │                                                    ▼
      ├─ 기존: 화재정보_대시보드.html / index.html → GitHub(midoseo/fire_dashboard) → Netlify 배포
      └─ 신규: fireDispatch.json ({national, bySido, liveItems}) → 같은 GitHub 커밋에 함께 업로드
                      │
                      ▼
      블루스캔 앱이 실행될 때 그 JSON을 fetch → 대시보드 스크롤 리스트 + 지도 🚒 마커에 표시 (실패 시 목업 폴백)
```

`liveItems`는 스케줄의 "2. 전국 실시간 화재정보(safekorea)" 단계에서 **이미 수집하고 있는 데이터**를 그대로
재사용하면 된다(좌표까지 이미 확보됨) — 새로 크롤링 범위를 늘릴 필요는 없고 JSON 저장 형식만 맞추면 된다.

### 중앙화 권장 — App.jsx에서 한 번만 fetch

지금 구조상 `Dash.jsx`(대시보드 카드), `ListA.jsx`/`ListB.jsx`(지도 마커)가 각각 독립적으로
`buildFireDispatchDemo()`를 호출한다. 목업 단계에선 문제없지만(순수 함수 호출), 실제 네트워크 fetch로
바뀌면 화면 전환마다 중복 호출이 생긴다. 권장 방식:

```js
// App.jsx — 최상단에서 한 번만
const [fireDispatch, setFireDispatch] = useState(() => buildFireDispatchDemo());
useEffect(() => {
  let alive = true;
  fetchFireDispatchLive().then(live => { if (alive && live) setFireDispatch(live); });
  return () => { alive = false; };
}, []);
```
그 다음 `<SalesDash .../>`, `<ListAScreen .../>`, `<ListBScreen .../>` 호출부에 `fireDispatch={fireDispatch}`를
prop으로 내려주고, 각 화면 내부의 `buildFireDispatchDemo()` 직접 호출을 지운다(대신 prop으로 받은 값 사용).
(`useState`, `useEffect`는 App.jsx에 이미 import되어 있음)

- Netlify가 `fireDispatch.json`을 정적 파일로 서빙할 때 CORS 헤더가 열려 있는지 (보통 Netlify 기본 정적 호스팅은 별도 설정 없이 공개 CORS 허용)
