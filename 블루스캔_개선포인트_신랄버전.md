# 블루스캔 타깃 공략 에이전트 — 냉정한 개선 포인트

> 관점: 영업관리 기획자 / 현장 영업담당자
> 작성 기준: `app/src` 실제 구현 + `PRD_v3_수정본.md` + `임원진_시연_시나리오.md` + 더미데이터 대조
> 톤: 의도적으로 신랄하게. "잘한 것"은 뺐고 "터질 것/약한 것"만 모았다.

## 0. 한 줄 총평

**데모는 예뻐 보이지만, PRD가 핵심 차별점이라고 주장한 "거절 자산화 → 재접촉 플라이휠"과 "이탈 스코어링"이 실제로는 비어 있다.** 지금 상태는 "영업 시스템"이 아니라 "잘 만든 영업 대시보드 목업"이다. 임원이 표면만 보면 통과하지만, 영업 실무자나 깐깐한 임원이 두 번째 질문("그래서 거절한 고객 다시 어떻게 챙기냐?", "이탈 위험 순위 어떻게 나오냐?", "새로고침하면 내가 오늘 입력한 게 남냐?")을 던지는 순간 흔들린다.

## 1. 시연 당일 실제로 터질 수 있는 것 (최우선)

### 1-1. "AI 분석"이 배포본에서는 AI가 아니다
- `Visit.jsx`의 `analyze()`는 3단계 폴백이다: ① `/api/analyze-visit` 백엔드 → ② `window.claude` → ③ 규칙 엔진(`analyzeBluescan`).
- 그런데 `/api/analyze-visit`는 **`vite.config.js`의 dev 미들웨어일 뿐**이다. GitHub Pages(정적 호스팅)에는 서버가 없다. `window.claude`도 배포본엔 없다.
- **결론: 데모 URL에서 "AI로 결과 분석"을 누르면 100% 규칙 엔진(정규식 매칭)이 돈다.** 임원이 "이거 진짜 LLM이에요?"라고 물으면 "네"라고 답하면 안 된다. 정직하게 "규칙 기반이고 실서비스에선 LLM 연동"이라고 해야 하는데, 그 순간 데모의 임팩트가 반감된다.
- 게다가 `SAMPLE_DIALOG`는 규칙 엔진이 화재·누수·설비·야간공백 키워드를 전부 물도록 **의도적으로 설계된 대본**이다. 임원이 "다른 내용으로 말해볼게요" 하고 마이크로 즉흥 발화하면 → (a) 크롬+마이크 권한+인터넷 필요, (b) 규칙 엔진이 엉뚱하게 매칭하거나 "정보 없음"을 뱉을 위험.

### 1-2. 날짜가 하드코딩되어 "오늘"이 오늘이 아니다
- 화재 데이터(`fireDispatch.demo.js`)의 `baseDate`는 `2026-07-04`, `LIVE_ITEMS` 시각은 전부 `07/04`.
- 대시보드는 이걸 **"오늘의 화재 출동 현황"**, **"실시간 화재 출동 현황"**으로 라벨링(`Dash.jsx`). 시연이 7/5 이후면 화면엔 "오늘의 화재 … 기준일 2026-07-04"라는 자기모순이 뜬다. "실시간"이라며 며칠 전 날짜가 박혀 있으면 신뢰도에 금이 간다.
- 유지고객 더미(`retention.demo.js`)는 `TODAY = 2026-07-04`로 못박고 계약종료일을 역산하는데, 화면의 `needsAttention()`은 **실제 `new Date()`(진짜 오늘)**로 D-day를 계산한다. 즉 **시연이 늦어질수록 D-day가 매일 1씩 밀린다.** 시나리오가 약속한 "한빛빌딩 D-44"는 7/5엔 D-44가 아닐 수 있고, 며칠 뒤엔 D-40대가 깨진다.
- 방문 저장(`Visit.jsx handleSave`)은 `date: '2026-06-10'` 하드코딩, CSV 파일명도 `방문결과기록_20260610.csv`. **오늘 입력한 기록에 6월 10일 날짜가 찍힌다.**

### 1-3. 세션 리셋 — 게이미피케이션의 자기부정
- 포인트·퀘스트·방문기록은 전부 세션 메모리 기반(`computeSessionPoints`는 "누적 저장 없음 — 데모 세션 기준"이라고 코드 주석에 명시). **새로고침 한 번이면 다 날아간다.**
- 시나리오도 "막히면 새로고침 후 재로그인(포인트·방문기록 초기화)"을 대비책으로 적어뒀다. 문제는 이게 데모 대비책이 아니라 **제품의 근본 한계**라는 점이다. 게이미피케이션의 핵심 동기(포인트가 "쌓인다")가 실제로는 매 세션 0으로 리셋된다. "실시간으로 쌓입니다"라는 멘트와 "새로고침하면 사라짐"이 정면충돌한다.

### 1-4. 외부 의존성이 곧 단일 장애점
- 지도 타일, 아이콘 폰트(Material Symbols), 화재/침수 오버레이가 전부 외부 로드. 시나리오도 "인터넷 필수"라고 못박음. 발표장 와이파이가 흔들리면 지도가 회색으로 뜨거나 아이콘이 텍스트("expand_more", "local_fire_department")로 노출된다 — 아주 없어 보이는 사고.
- `appData.generated.js`가 **5.8MB 단일 번들**, HTML 배포본은 6.8MB. 현장 LTE/모바일 첫 로딩이 무겁다. "모바일에서도 동일하게 작동"이라는 마무리 멘트의 리스크.

## 2. PRD가 "Must"라던 기능이 실제로는 없다 (정합성 갭)

시나리오 부록은 이걸 "의도적 미구현"으로 포장했지만, 영업기획 관점에선 **차별점 그 자체가 빠진 것**이다.

| PRD | 우선순위 | 상태 | 냉정한 평가 |
| --- | --- | --- | --- |
| R16 거절 사유(REJ-01~06) 필수 구조화 | **Must** | ❌ 없음 | 방문상태에 '거절' 버튼만 있고 사유 선택 UI가 없다. `demoVisits.js`에 거절 메모 텍스트 1건뿐. |
| R17 사유별 재접촉 예정일 자동 산출 + 영업 알림 | **Must** | ❌ 없음 | PRD 표(예산 D-30, 경비원 만료 D-60 등)가 통째로 미구현. |
| R18 재접촉 알림 클릭 → 히스토리+AI요약+거절사유 한 화면 | Should | ❌ 없음 | — |
| R20 유사 레퍼런스 자동 매칭 | Should | ❌ 없음 | 코드에 흔적조차 없음. 영업의 강력한 "근거 무기"가 빠짐. |
| R8 타깃 검토·승인 → 확정 리스트 반영 | **Must** | △ 부실 | 승인 워크플로 없음. '확정 리스트'는 그냥 방문 입력한 것 모음. |
| R19 분기 전환율 리포트(가중치 보정) | Could | ❌ 없음 | "플라이휠" 서사의 검증 루프가 없음. |
| R22 이탈 위험 구간별 해약 방어 시나리오 | **Must (테마B)** | ❌ 없음 | 생애가치 박스(BEP/ROI)만 있고, "그래서 뭘 하라"는 시나리오가 없음. |
| 이탈 스코어링(테마 B 핵심) | Must | ❌ 없음 | "주의 필요"는 단순 OR 규칙(D-60 or 심각신호 or VOC>0). 스코어·순위·변별력 없음. |
| R24 관제신호 → VOC 접수 전 선제 분류 | Should | △ 흉내 | `unresolvedVOC` 숫자만 표시, 자동 분류/알림 기능 아님. |

**핵심 타격 포인트:** 거절 자산화(R16/R17)는 이 제품이 "세콤/타사 대비 왜 다른가"의 논거였다. PRD 서론에서 "거절이 버려진다"를 문제로 정의해놓고, 정작 그걸 푸는 기능을 안 만들었다. 게이미피케이션은 심지어 "거절(사유 기록) +5P"를 배점(`gamification.js POINTS.visitReject`)해놨는데 **사유 기록 UI 자체가 없다.** 앞뒤가 안 맞는다.

## 3. 데이터/신뢰성 — 깐깐한 임원이 파고들 지점

### 3-1. 더미데이터가 "생성된 티"가 난다
- `retention.demo.js`는 순환 배열로 찍어낸다: 월서비스료 10종(`MONTHLY_FEE_CYCLE`), 공사비 10종, 할인율 10종, 계약개월 10종… **40개 고객이 사실상 4세트 반복.** 상세를 여러 개 펼치면 같은 숫자가 반복되는 걸 들킨다.
- 신호이력·VOC·원격제어가 전부 `i % n` 기반 → 패턴이 규칙적이라 "실데이터 아니죠?"에 방어가 약하다.
- 40개 계약처 전부 **마포구 12개 동만 순환**. 실제 지사 포트폴리오처럼 보이지 않는다(한 동에 3~4개씩 몰림).

### 3-2. BEP/ROI 로직이 발표의 하이라이트인데 검증이 얇다
- 시나리오 핵심 멘트가 "지금 해약되면 회사가 손해 → 할인해서라도 잡아라"인데, 이 판단은 `netValueToDate < 0` 하나로 갈린다. **"3년 기준"이라 표기하면서 현재 누적값으로 판정**하는 로직의 정합성(공사비·월료·경과개월·할인율의 관계)을 임원이 계산기로 두들기면 설명이 궁색해질 수 있다. 발표의 클라이맥스일수록 숫자 근거를 방탄으로 만들어야 한다.

### 3-3. 리더보드/동료 점수가 가짜
- 동료 점수는 사번 해시 시드(`seededPoints`)로 만든 결정론적 난수. 본인 점수만 실제. 부록에 적어뒀지만, "랭킹으로 경쟁 유도"가 셀링포인트인데 경쟁 상대가 허수라는 건 약점이다.

### 3-4. 개인정보/실명 노출 관성
- `retention.demo.js` 컨설턴트 목록에 실명(`서지나` 등)이 박혀 있다. 시연·배포 전 비식별 점검 필요. 월간 리포트는 인근 계약처를 동 단위로 익명화했지만(잘함), 화면 다른 곳의 실명·계약번호는 그대로 노출된다.

## 4. 기능 자체의 영업 실무 관점 약점

- **유지고객 대시보드에서 "행동"을 할 수 없다.** 리포트 보내기·감성터칭 외에 전화 로그, 방문 예약, 해약 방어 액션 기록이 없다. 위험을 보여주기만 하고 "그래서 뭘 클릭하면 되는지"가 없다. 영업은 "다음 행동"이 없으면 안 쓴다.
- **"주의 필요"가 남발된다.** D-60 이내 or 심각신호 or VOC>0 중 하나만 걸려도 빨간 배지. 마포 40곳 중 상당수가 주의필요로 떠서 **우선순위 변별력이 사라진다.** 진짜 스코어링이 필요한 이유가 바로 이것인데 그게 없다.
- **감성터칭/월간리포트 "발송(모의)"이 끝나면 아무 일도 안 일어난다.** "그래서 실제로 카톡으로 나가요, 메일로 나가요?"에 대한 답(채널·정책)이 PRD Open Question에 미결로 남아 있음. 데모에선 상태 텍스트만 바뀜 → "이거 진짜 나가는 거 아니죠?" 질문에 취약.
- **거절 없는 방문 퍼널.** "방문 → 거절 → 재접촉 → 전환" 퍼널을 측정하겠다는 성공기준이 있으나, 거절 사유가 안 쌓이니 퍼널 측정 자체가 불가능한 구조.
- **신규(리스트 A)와 유지(테마 B)의 연결이 서사뿐.** "데이터 공유 플라이휠"이라 했지만 실제로 두 화면은 데이터·화면이 분리돼 있고 서로를 강화하는 동작이 코드에 없다.

## 5. UI/UX 신랄 포인트

- **고정 픽셀 폭 모달 = 모바일에서 깨질 위험.** 방문입력 `width:1020`, 월간리포트 `760`, 감성터칭 `520`. "모바일에서도 동일"이라는 멘트와 배치된다. 좁은 화면에서 좌우 2컬럼(폼+타임라인, 목록+지도)이 어떻게 접히는지 반드시 실기기 확인 필요.
- **"주의 필요" 외에 색·배지 위계가 약하다.** 심각/주의/경미, 화재/침수, 등급 S~D, 상품 듀얼/오너 배지가 한 행에 몰리면 정보 과밀 → 무엇부터 볼지 안 잡힌다.
- **드릴다운 부재.** KPI 카드(전체/주의/만료임박)를 눌러도 해당 목록으로 필터링되지 않는다. 숫자만 있고 인터랙션이 없다.
- **아이콘 폰트 로드 실패 시 텍스트 노출.** 위 1-4와 연결 — 네트워크 지연 시 `expand_more` 같은 raw 문자열이 잠깐 뜬다.
- **PDF가 이미지 캡처(html2canvas).** 텍스트 선택·검색 불가, 용량 큼, 한글 폰트/스케일 깨질 여지. "고객사에 바로 보낼 완성본"이라기엔 품질 리스크. 포털로 띄운 다이얼로그 캡처는 스크롤/여백 잘림도 잦다.
- **정렬/필터가 지도와 완전 동기화되는지 체감이 약함.** 검색·필터 시 지도 fit이 튀거나 선택 상태가 어긋날 수 있어(대량 마커) 실기기 확인 권장.

## 6. 우선순위 제안 (냉정하게 순서대로)

**시연 전 반드시 (반나절~1일):**
1. 화재 `baseDate`/`LIVE_ITEMS` 날짜, `retention.demo`의 `TODAY`, `Visit` 저장 날짜를 **실행 시점 기준 동적 계산**으로 교체. "오늘"이 진짜 오늘이 되게.
2. "AI 분석" 문구를 정직하게 — 배포본은 규칙 기반임을 인정하거나, 최소 1개 시나리오라도 실제 LLM(백엔드 키)으로 붙여 "진짜"를 하나 확보.
3. 세션 리셋 문제에 대한 방어 멘트/최소 localStorage 지속화 검토. 최소한 발표 중엔 새로고침 금지 동선.

**차별점 복구 (1~2주, 여기서 승부):**
4. **거절 사유(R16) + 재접촉 자동 알림(R17)** 최소 구현. 이게 없으면 "왜 우리 제품인가"가 사라진다. 데모 한 케이스만이라도 "거절 → 30일 뒤 재접촉 알림 → 재방문" 루프를 보여줘라.
5. **이탈 스코어(테마 B 핵심) + 해약 방어 시나리오(R22)** 최소 구현. "주의 필요" 남발을 스코어 순위로 대체.
6. **유사 레퍼런스 매칭(R20)** — 영업의 근거 무기. 규칙 매칭만으로도 임팩트 큼.

**신뢰도 (병행):**
7. 더미데이터 순환 티 제거(값에 노이즈/편차 부여), 마포 편중 완화, 실명 비식별.
8. BEP/ROI 계산식 문서화 + 숫자 방탄 검증.

**UX (후순위):**
9. KPI 드릴다운, 배지 위계 정리, 모바일 실기기 QA, PDF 텍스트 기반 전환 검토.

## 7. 임원이 던질 킬러 질문 (미리 대비할 것)

1. "이거 진짜 AI가 분석하는 거예요, 규칙이에요?" → (현재 배포본은 규칙)
2. "거절한 고객은 어떻게 다시 챙겨요?" → (기능 없음)
3. "이탈 위험 1순위 고객이 누구예요? 순위 어떻게 매겨요?" → (스코어 없음)
4. "제가 오늘 입력한 거 내일도 남아 있어요?" → (세션 리셋)
5. "감성터칭 메시지 실제로 어디로 나가요?" → (채널 미정)
6. "이 숫자들 실데이터예요?" → (생성 더미, 순환 패턴)
7. "모바일에서도 이대로 돼요?" → (고정폭 모달, 실기기 미검증)

이 7개에 대한 답이 준비되면 시연은 방어된다. 답이 궁하면 그 기능이 다음 스프린트 1순위다.
